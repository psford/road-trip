# Map POI & Enhancements Design

## Summary

Road trip is a photo map application where users post photos and pin them to geographic locations on a map. This feature adds Points of Interest (POIs) — national parks, state parks, historic sites, natural features, and tourism landmarks — as interactive map markers that make location pinning faster and more accurate. Rather than requiring users to drop a pin by manually tapping an approximate location, they can tap a labeled POI marker and either adopt its exact coordinates or zoom in for a precise nearby spot.

The implementation uses a hybrid approach: park boundary polygons are rendered directly from MapTiler's existing vector tile data (with visual style overrides to make them more prominent), while interactive POI markers are stored in the application's own database and served through a new viewport-scoped API endpoint. A one-time seed script populates the database from three free public sources (NPS API, PAD-US, and OpenStreetMap via Overpass), with idempotent re-runs supported for periodic refresh. Markers load on demand as the user pans and zooms, filtered by category at each zoom level to avoid clutter on wide-area views.

## Definition of Done

1. **POI markers on both post and view maps** — national/state parks, historic sites, natural features, and tourism POIs displayed as prominent, labeled markers sourced from our DB
2. **Park polygon restyling** — MapTiler vector tile styles overridden so park boundaries are visually prominent (bolder green fill, labels at lower zoom levels)
3. **Tap-to-pin interaction on post page** — tapping a POI marker offers "Use this location" (exact POI coords) or "Pick nearby spot" (zoom in to refine). View page shows markers but no tap-to-pin interaction
4. **Regional loading** — POIs loaded by geographic region so users on low bandwidth don't download the full US dataset
5. **POI data pipeline** — a seeding/import process to populate the DB from free sources (PAD-US, Overpass, NPS — exact mix TBD)
6. **Pin-drop map defaults to last photo** — when opening the pin-drop UI, map centers on the last posted photo's location instead of center of US

## Glossary

- **POI (Point of Interest)**: A named geographic location worth displaying on a map — parks, historic sites, natural landmarks, tourism attractions. Stored as centroid lat/lng coordinates in the application database and displayed as interactive markers.
- **MapTiler**: A commercial map tile provider. The road-trip app uses MapTiler's vector tile styles as its base map. Park boundary polygons are part of MapTiler's tile data and are restyled rather than re-sourced.
- **MapLibre GL JS**: An open-source JavaScript mapping library (fork of Mapbox GL JS) used to render vector tiles and custom layers in the browser.
- **Vector tiles**: Map data encoded as geometric features (polygons, lines, points) rather than raster images. Allows client-side style overrides at any zoom level without re-fetching data.
- **GeoJSON**: A standard JSON format for geographic data. Used here as the data format for POI markers fed into MapLibre's layer system.
- **Source-layer**: A named subset of data within a vector tile set. `setPaintProperty()` targets a specific source-layer (e.g., `landuse`, `park`) to override default MapTiler styles.
- **PAD-US**: Protected Areas Database of the United States — a free federal dataset (USGS) listing protected lands including state parks and conservation areas, available as downloadable GeoJSON.
- **NPS API**: The National Park Service's public REST API, providing data on ~470 NPS-managed sites (national parks, monuments, battlefields, seashores, etc.).
- **Overpass API**: A read-only public API for querying OpenStreetMap data. Used here to extract tourism, historic, and natural feature nodes within the US.
- **Centroid**: The geographic center point of a polygon (e.g., the center of a park boundary). POIs are stored and displayed as centroids rather than full polygons.
- **Upsert**: A database write that inserts a new row if the record does not exist, or updates it if it does — used to make the seed script idempotent.
- **Pin-drop**: The road-trip app's UI gesture for assigning a geographic location to a photo — the user taps the map to place a pin at the desired coordinates.
- **Debounce**: A technique that delays execution of a function until a pause in rapid events — here applied to the `moveend` map event so the API is not called on every pixel of a pan.

## Architecture

Hybrid approach: MapTiler vector tiles provide park boundary polygon rendering (restyled for prominence), while our database stores POI centroid points for interactive markers and labels.

**Data flow:**
1. CLI seed script imports POI data from PAD-US, Overpass API, and NPS API into `roadtrip.PointsOfInterest` table
2. Frontend calls `GET /api/poi?minLat=X&maxLat=Y&minLng=Z&maxLng=W&zoom=N` on map viewport changes (debounced)
3. Backend queries Azure SQL with bounding-box filter and zoom-based category filtering, returns up to 200 POIs
4. Frontend renders POIs as a GeoJSON overlay layer on the MapLibre map
5. On the post page, clicking a POI marker shows a popup with "Use this location" / "Pick nearby spot" options

**Key components:**

- **`roadtrip.PointsOfInterest` table** — stores POI name, category, lat/lng centroid, data source, and external source ID for dedup
- **`GET /api/poi` endpoint** — viewport-based query with zoom-dependent category filtering and result cap
- **`PoiSeeder` CLI tool** — standalone script that fetches from external sources, deduplicates, and bulk upserts
- **MapTiler style overrides** — `setPaintProperty()` calls on `landuse` and `park` source-layers after map load to increase fill opacity, add outlines, and add park labels at lower zoom levels
- **POI GeoJSON overlay** — `geojson` source + `circle` layer (markers) + `symbol` layer (labels) added to MapLibre map, updated on viewport change
- **Tap-to-pin popup** — post page click handler on POI markers offering location selection or zoom-to-refine

**Pin-drop map default location:** When pin-drop map opens, check if trip has existing photos. If yes, center on last photo's coordinates at zoom 10. If no, fall back to center of US at zoom 4.

### API Contract

**`GET /api/poi`**

Query parameters:
- `minLat`, `maxLat`, `minLng`, `maxLng` (required, float) — viewport bounding box
- `zoom` (required, int) — current map zoom level

Response: `PoiResponse[]`
```json
[
  {
    "id": 42,
    "name": "Franconia Notch State Park",
    "category": "state_park",
    "lat": 44.1427,
    "lng": -71.6814
  }
]
```

Category filtering by zoom:
- zoom < 7: `national_park` only
- zoom 7-9: adds `state_park`, `natural_feature`
- zoom 10+: adds `historic_site`, `tourism`

Result cap: 200 per request. No auth required (public reference data).

### Database Schema

**`roadtrip.PointsOfInterest`**

| Column | Type | Constraint |
|--------|------|-----------|
| `Id` | int | PK, auto-increment |
| `Name` | nvarchar(300) | required |
| `Category` | nvarchar(50) | required — `national_park`, `state_park`, `historic_site`, `natural_feature`, `tourism` |
| `Latitude` | float | required |
| `Longitude` | float | required |
| `Source` | nvarchar(50) | required — `pad_us`, `osm`, `nps` |
| `SourceId` | nvarchar(200) | nullable, indexed — external ID for upsert dedup |

**Indexes:**
- Composite index on `(Latitude, Longitude)` for bounding-box queries
- Index on `SourceId` for idempotent re-import
- Index on `Category` for zoom-based filtering

## Existing Patterns

**EF Core entity + migration pattern:** Existing entities (`TripEntity`, `PhotoEntity`, `GeoCacheEntity`) in `src/RoadTripMap/Entities/` with DbContext configuration in `src/RoadTripMap/Data/RoadTripDbContext.cs`. New `PoiEntity` follows this pattern. Schema is `roadtrip` (configured in `OnModelCreating`).

**Minimal API endpoints:** All endpoints defined in `src/RoadTripMap/Program.cs` as `app.MapGet()`/`app.MapPost()`. New POI endpoint follows this pattern.

**GeoCache as reference data precedent:** `roadtrip.GeoCache` is the closest existing pattern to POI data — non-user-created, populated by external source (Nominatim), queried by coordinates. POI table follows the same concept but populated by seed script rather than on-demand.

**Frontend JS module pattern:** Each concern has its own file in `src/RoadTripMap/wwwroot/js/` (e.g., `mapService.js`, `postUI.js`, `photoCarousel.js`). New POI layer logic follows this pattern as a new module (e.g., `poiLayer.js`).

**MapLibre map initialization:** Both post page (`postUI.js:409-445`) and view page use the same `MAP_STYLE` constant and map init pattern. Style overrides and POI layer setup hook into the existing `map.on('load')` handler.

## Additional Considerations

**POI data volume:** Expected ~5,000-15,000 POIs across all categories and sources. At ~100 bytes per row, this is well under 5MB — negligible relative to the 2GB database capacity.

**Low-bandwidth optimization:** Viewport-based loading with 200-result cap keeps individual API responses small (~20KB). Debounced `moveend` prevents rapid-fire requests during pan/zoom.

**Data refresh:** Re-run the seed script periodically (quarterly) to pick up new parks/sites. No automated schedule — manual is sufficient for reference data that changes infrequently.

**Dedup across sources:** A national park may appear in both NPS API and PAD-US. Deduplication uses name similarity + proximity (~1km threshold). `SourceId` ensures re-runs of the same source don't create duplicates; cross-source dedup uses the proximity check.
